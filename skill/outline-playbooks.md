# 개요 유형화 플레이북 + 검증 게이트 (writing-assistant 차용판)

> **출처**: adapted from an internal writing-assistant toolkit — 개요→승인→초안 루프, 장르 플레이북, 4렌즈 검토를 the-author 4유형에 맞게 번안. **이 toolkit은 the author 본인이 만든 글쓰기 코치 시스템이 원본** — 즉 아래 규칙은 외부 folklore가 아니라 저자 자신의 작법론이다(저자 문체를 재현하는 스킬이 저자의 작법 위에서 생성해야 하는 이유).
> **원칙**: 개요 유형화 = 글쓰기의 기본. **개요 승인 전에는 본문을 쓰지 않는다.**
> **위계**: 문체는 마지막 광택이 아니라 초안 단계부터 구조·리듬에 관여한다 — 단 **목적·독자·장르 규칙이 항상 문체보다 위**다. 재현 장치가 전달을 해치면 장치를 버린다(배치 내 장치 반복·기계적 CTA가 판정단 tell이 된 R1 실측 근거).

## 0. 집필 파이프라인 (SKILL.md 집필 단계와 결합)

```
유형 판정 → 레지스트리 앵커 선정 → [G1] 스켈레톤 사상 개요(실글 앵커 1차 · 주형 fallback) → [G2] 개요 승인 게이트
  → 차용 집필(기존 2~3단계) → [G3] 4렌즈 검토 → 발신 게이트(check_endings + band_scorer + structure_scorer + gate.py)
```

- **G1-A 스켈레톤 앵커 (v2 — 1차 경로)**: 개요는 유형 주형이 아니라 **실글에서 출발**한다. `registry.py pull`로 같은 장르 실글 1~2편(grade 우선·substance low 제외)을 뽑고, `skeleton_extract.py`가 낸 슬롯 맵(훅/전개/전환/마무리 subtype)에 **전하려는 메시지를 사상**한다. 개요 frontmatter에 `skeleton_anchor: <실글 ref> — <슬롯 대응>` 명시 의무 — 앵커 없는 개요는 초안이 아니라 버그. 배치 작업에선 앵커·subtype을 회전(다양성 가드 — 한 앵커 기계 반복 = 그 자체가 AI 티). 아래 §1~4 주형은 **앵커를 못 구했을 때의 fallback**이며, fallback 사용 시 개요에 그 사실을 적는다.
- **G1 개요 생성 — 새 글 작성 시엔 "함께 짜기" (author policy)**: 새 글(신규 집필)은 개요를 봇이 완성해 통보 ❌ → **the author과 인터랙티브로 확정**: ①**방향안 2~3개 제시**(각 방향의 핵심 메시지 1줄 + 추천 1개 — writing-assistant ideation 차용) ②the author 선택/수정 ③선택된 방향으로 유형별 주형을 채운 개요 초안(`/prompt` 경유) ④the author 다듬기 반영. 재작성/윤문 글은 기존 G1(개요 산출→G2)로 충분. 개요는 목차가 아니다 — 최소 포함: **제목/훅 후보 2~3 · 핵심 메시지 1줄 · 독자에게 주는 약속 1줄 · 블록 구조(유형 주형) · 블록별 예상 분량 · 필요한 실경험/근거 · 아직 빈 곳**.
- **G1.5 "잘 쓰는 법" 반영 체크**: 개요 초안엔 해당 유형 주형의 **"좋은 시작" 방식 중 무엇을 쓰는지**와 **"흔한 실패" 회피 확인**(§1~4 각 표의 실패 행)을 1줄씩 명시 — 주형을 로드만 하고 안 쓰는 드리프트 차단.
- **G2 개요 승인 게이트 (hard)**: the author 의뢰 글 = the author 개요 확인 후 본문. 자율 산출 글 = 개요를 채팅에 먼저 제시하고 명시 진행 신호 후 본문. **개요 없이 본문 직행 = 위반** (writing-assistant 절대원칙 5·6 차용).
- **G3 4렌즈 검토 (초안 후, 발신 게이트 전)**: 아래 §5. 문장 다듬기는 마지막.

## 1. 사색/논평 개요 주형 (에세이 플레이북 번안)

| 블록 | 내용 | 스레드 분량 감각 |
|---|---|---|
| ① 장면/균열 | 구체 장면·작은 불편함으로 연다(추상 명제 ❌) | 1~2줄 |
| ② 경험/관찰 | 무슨 일이 있었나 — 1인칭 평어 | 2~4줄 |
| ③ 질문 | 그 경험이 던진 질문 한 개 | 1줄 |
| ④ 해석/통찰 | 단정으로 민다("~거다") — 교훈조 ❌ | 2~3줄 |
| ⑤ 여운 | 짧은 단정 or 명사 종결 or "(아님)" 자조 | 1줄 |

흔한 실패(차용): 일기처럼 흘러감 · 통찰 없이 감정만 · 도입 산만.

## 2. 정보/튜토리얼 개요 주형 (블로그/보고서 플레이북 번안)

| 블록 | 내용 | 분량 감각 |
|---|---|---|
| ① 독자 효용 훅 | "이 글 끝나면 ~가 된다" — 초반에 효용 | 1~2줄 |
| ② 전제/맥락 | 필요한 배경 1개만(과잉 설명 ❌) | 1~2줄 |
| ③ 단계/방법 | `→` 단계 구조, 블록당 하나 · 실제 사례(§F 편집 패턴: 실경험·수치·화면) | 3~6줄 |
| ④ 함정/팁 | 흔한 실패 1개 + 회피법 — 자조 섞기 | 1~2줄 |
| ⑤ 다음 예고/정리 | 연재면 다음 편 예고, 단발이면 1줄 정리 | 1줄 |

합니다체 일관(§E 혼용 금지) — 개요 단계에서 어미 노선까지 확정.

## 3. 홍보/소개 개요 주형 (뉴스레터/칼럼 번안)

| 블록 | 내용 |
|---|---|
| ① 경험 도입 | 평어 — "~것이었다" 실경험으로 연다(광고 문구 ❌) |
| ② 무엇인가 | 합니다체 정보 — 팩트만 |
| ③ 이해상충 정직 | "강의 팔지만/제 제품이지만" 날것 고백 |
| ④ 행동 유도 | 친구 톤("무료일 때 ㄱㄱ") + 명사 종결 |

## 4. 후기/공유 개요 주형

| 블록 | 내용 |
|---|---|
| ① 계기 | 왜 갔나/왜 썼나 — 평어 1줄 |
| ② 인상 2~3개 | 구체 장면·인용 위주(`''`·`@멘션` 출처) |
| ③ 내 것으로 | 내 작업/생각에 어떻게 붙는지 1개 |
| ④ 여운/추천 | 짧은 단정 |

## 5. G3 — 4렌즈 검토 (review-lenses 차용, 초안 → 발신 게이트 사이)

순서 고정: 목적 → 구조 → 근거 → 독자 → (마지막에만) 문장.

1. **Structural**: 핵심 메시지가 한 문장으로 보이나 · 도입이 일하나 · 결론에서 새 주장 추가 ❌.
2. **Reader**: 어디서 멈추나 · "그래서 나랑 무슨 상관"에 답하나 · 효용이 초반에 보이나.
3. **Skeptic/Fact**: 사실·해석·추정 분리 — 모르는 것을 사실처럼 ❌ · 수치·인용 출처.
4. **Voice**: 유형별 문체(A~E) 대조 — 이건 기존 4단계와 동일, **가장 마지막**.

각 렌즈는 "다 고쳐주기"가 아니라 **무엇을 먼저 고칠지 지목**(차용 원칙). 렌즈 통과 후에만 발신 게이트(check_endings·gate.py)로.

## 6. 역개요 모드 — 기존 글 개작·재작업 경로 (작법 SoT 차용)

이미 써둔 글(초안·타인 글·과거 글)을 받으면 **무조건 처음부터 다시 쓰지 않는다.**

1. **역개요 먼저**: 문단별 역할 1줄 요약 → 중복 파트·연결 약한 곳·순서를 바꾸면 좋아질 곳 표시 (`skeleton_extract.py`의 슬롯 맵을 개작 진단에 그대로 씀 — 생성용 도구와 동일).
2. **수선 범위 지목**: 구조 재설계 / 논리 보강 / 문장 정리 / 문체 조정 / 요약·확장 / 장르 변환 중 무엇이 필요한지 좁힌 뒤 그것만 손댄다.
3. 퇴고 순서는 §5와 동일(목적→구조→근거→독자→문장·문체 마지막) — 초안 뒤 자동 문장 광택부터 시작 ❌.

## 7. 전역 안티패턴 (작법 SoT 차용 — 유형 불문)

- 자료(레지스트리 실글·차용 소스)를 안 보고 감으로 초안 쓰기.
- 문체 흉내에 집착해 목적 전달을 망치기 — 위계(헤더) 위반.
- 초안 뒤 자동으로 문장만 예쁘게 다듬기 (퇴고 범위 지목이 먼저).
- 요청받지 않은 "AI스럽게 친절한 군더더기" 붙이기 (완결형 리캡·안내 문구 — R1 tell 사유와 동형).
- 사실 검증이 필요한 주장을 과감하게 단정하기 (§5 Skeptic 렌즈로 회수).

## 차용하지 않은 것 (경계 기록)

- 10차원 Style Card: 우리는 the canonical style profile+코퍼스가 그 역할(이중 SoT 방지) — 도입 ❌.
- 6장르 전체: 스레드 4유형에 맞게 번안만. 얼룩소 장문(v5 트랙)에 확장할 땐 에세이/칼럼 주형 원본 참조.
- persona_carve 루프: 우리 corpus 파이프라인(build_corpus.py)이 대체.

> 과경화 가드(skill-process §6): 주형은 판정·출발점이지 기계 주형 ❌ — 실제 글 차용(기존 2~3단계)이 여전히 본체. the author 최종 피드백 우선.
